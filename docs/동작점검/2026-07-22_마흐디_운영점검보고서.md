## 6. 고도화 방안 (실행 점검 결과 기반)

1. **기동·종료 생애주기를 완전히 대칭적으로 idempotent하게 만들 것** — 어제는 "종료 시 잔존 프로세스"만 다뤘는데, 오늘 조사에서 "기동 시에도 잔존 프로세스를 먼저 치워야 한다"는 대칭적 갭이 드러났다. Fix #1과 연결되는 구조적 원칙으로 `docs/dev_memory/DECISION_LOG.md`에 명문화 권장 — "COCKPIT/관측 루프를 띄우는 모든 진입점(수동 재시작 포함)은 시작 전 자기 자신의 잔존 인스턴스를 먼저 제거한다."
2. **레이트리밋 근접도를 COCKPIT에 실시간 노출** — 사후 로그 분석 없이 "오늘 레이트리밋에 얼마나 근접했는지"(예: 초당 사용량/한도 비율, 최근 1시간 실패율)를 "오늘의 점검 요약" 배지에 추가하면 §2-1 같은 재발을 다음날 로그 정밀분석 없이도 그날 바로 포착 가능.
3. **로그 회전 경계를 항상 명시적으로 확인하는 습관 재확인** — 어제 보고서도 §2-2에서 한 번 겪었던 "회전 파일에 이전 날짜 내용이 섞여 오독" 문제가, 오늘은 정반대 방향(cockpit.log가 회전이 안 돼 누적)으로 다시 나타났다. 두 로그 파일의 회전 방식이 다르다(observation_loop.log는 Python RotatingFileHandler, cockpit.log는 배치스크립트의 10MB 크기 체크)는 걸 점검자가 매번 기억해야 하는 구조 자체가 실수를 유발하기 쉬우므로, 장기적으로는 두 로그 모두 "이 프로세스가 언제 시작됐는지"를 자체 기록하게 만들어(Fix #3) 회전 방식과 무관하게 경계를 파일 내용만으로 판별 가능하게 하는 게 근본 해법이다.
4. **"우연히 맞는 값"을 신뢰 근거로 삼지 않는 습관** — §4의 messiah 발견 사례를 `KIS_RAW_FIELD_RANGES.md`류 체크리스트에 한 줄 추가할 만하다: 마스터파일 등 외부 데이터의 필드를 새로 읽을 때는 "정렬이 우연히 맞아떨어지는 경우"를 배제하기 위해 최소 한 번은 의도적으로 순서를 흩뜨려 재검증하는 절차를 권장.
