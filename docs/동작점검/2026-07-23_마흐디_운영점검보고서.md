## 5. 고도화 방안 (실행 점검 결과 기반)

1. **레이트리미터를 블랙박스에서 관측 가능한 컴포넌트로 전환** — Fix #1·#2와 직결. 지금은 파라미터 하나를 바꿀 때마다 "다음날 하루치 결과를 보고 사후 추정"하는 것 외에 검증 수단이 없다. 상태 전이 로깅이 들어오면 같은 날 안에도 "지금 몇 배 백오프 중인지"를 알 수 있어, 파라미터 튜닝의 피드백 루프가 하루(다음날 아침)에서 실시간으로 단축된다.
2. **"거의 같은 시각 재현" 패턴에 대한 관찰 습관화** — §2-3처럼 이틀 연속 5초 오차 내로 같은 사고가 재현된 사례는 단순 우연 판정 전에 최소 2~3일 더 관찰하는 것을 점검 루틴에 명시적으로 포함할 만하다. 하루치 로그만 보면 "우연"으로 넘기기 쉽지만, 여러 날을 겹쳐 보면 숨은 정기 패턴(외부 서비스의 스케줄, 내부 폴링 주기끼리의 우연한 정렬 등)이 드러날 수 있다.
3. **"기록 없음" 같은 정상 초기 상태 메시지와 실제 이상 상태를 구분하는 습관 재확인** — §1-3에서 "직전 COCKPIT 기동 기록 없음"을 보고 처음엔 마커 기능이 실패한 것으로 오판할 뻔했다(07-22 밤 늦게 커밋된 기능이라 그날은 실행 기회가 없었을 뿐). 신규 기능이 처음 실행되는 날은 "기록 없음"류 메시지가 버그가 아니라 정상 최초 실행의 증거라는 점을 점검 시 항상 먼저 확인할 것 — 07-22 report §6-3(로그 회전 오독 주의)과 같은 계열의 함정.
4. **WS 무결점 하루(§2-2)를 "새 기준선"으로 문서화** — 지금까지 매 보고서가 최소 1건의 WS 재연결을 전제로 서술해왔는데, 어제 처음으로 0건을 기록했다. 다음 재연결이 발생했을 때 "이게 흔한 일인지 이례적인 일인지" 판단하려면 이 무결점 기록을 기준값으로 남겨둘 필요가 있다(`DECISION_LOG` 또는 `CURRENT_STATE`에 한 줄 기록 권장).
