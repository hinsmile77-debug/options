## [2026-07-20] 배치파일 자기점검(직전 정상 실행 경과)은 Python으로 구현 — cmd.exe엔 날짜 연산/문자열 로직을 추가하지 않는다

**결정:** "직전 정상 실행 이후 경과 시간"을 로그에 남기는 기능을, `scripts/start_mahdi_premarket.bat`/
`stop_mahdi_marketclose.bat` 안에 cmd.exe 문법(날짜 비교, 마커 파일 읽기/쓰기, 한글 포매팅)으로
직접 구현하지 않고 전부 Python으로 옮겼다. `mahdi/main.py`에 `_log_startup_gap_since_last_run()`
(관측 루프 기동 시 직전 정상 기동 경과를 `observation_loop.log`에 기록)을, 신규
`scripts/log_marketclose_stop.py`(장마감 시 직전 정상 종료 경과를 `premarket_startup.log`에
기록)를 추가했다. 배치파일 쪽 변경은 `stop_mahdi_marketclose.bat`에 이 스크립트를 호출하는
`cd /d "%PROJECT_DIR%"` + `uv run python scripts\log_marketclose_stop.py` 두 줄뿐 — 새 REM
설명 블록도, 새 goto 라벨도, 새 `set /p` 읽기도 추가하지 않았다.

**Why:** 원안(cmd.exe 안에서 `if not exist`/`goto`/`set /p`로 마커 파일을 읽고 경과를 계산)을
먼저 구현해 스크래치패드에서 격리 테스트했는데, 새로 추가한 REM 주석·goto 라벨이 있는 블록에서
cmd.exe가 한글 텍스트를 예측 못한 방식으로 잘못 토큰화해(`'MARKER'`, `'art_check_skip'` 같은
라벨/변수명 파편이 "인식할 수 없는 명령"으로 실행 시도됨) 반복적으로 재현됐다. `chcp 65001` 유무,
`>>`/`<`/`&`/`|` 같은 리다이렉트 기호 제거 등 여러 가설로 원인을 좁혀보려 했지만 완전히 재현 가능한
규칙을 찾지 못했고, 그중 한 번은 `set /p ... <"파일"` 조합이 존재하지 않는 파일을 대상으로 실행되며
콘솔이 완전히 멈추는 사고까지 재현했다(다행히 격리된 스크래치패드 테스트였고, 라이브 프로세스에는
영향이 없어 즉시 강제 종료함). 이 프로젝트의 배치파일은 이미 한 번(2026-07-07) 미이스케이프 괄호로
인한 cmd.exe 파싱 버그를 실제 라이브 스케줄에서 겪은 전례가 있다 — cmd.exe의 한글/특수문자 처리는
이 코드베이스에서 반복적으로 신뢰할 수 없는 것으로 확인된 영역이므로, 그 위에 로직을 더 쌓기보다
로직 자체를 통째로 옮기는 쪽을 택했다.

**How to apply:** 앞으로 장전/장마감 배치파일에 새 기능이 필요하면, 배치파일은 "이미 검증된 단순한
줄 하나(프로세스 실행/종료, 파일 리다이렉트, 이미 쓰이던 패턴의 반복)"만 담당하게 하고, 조건 분기가
얽히거나 날짜·문자열 처리가 필요한 로직은 반드시 별도 Python 스크립트로 분리해 pytest로 테스트할
것. 배치파일을 수정했다면 실제 스케줄러 실행 전에 반드시 스크래치패드에서 격리 실행해 콘솔 출력을
직접 확인할 것(이번처럼 Docker/COCKPIT 기동 같은 핵심 동작은 살아있어도 부수 로직만 조용히 깨질 수
있다) — 그리고 무언가 "멈춘 것처럼 보이면" 원인을 더 파기 전에 즉시 프로세스를 강제 종료하고
안전한 대안으로 우회할 것(이번처럼).

## [2026-07-20] KIS 모의투자 레이트리밋 재실측(2건/초 → 1건/초) — 적응형 백오프로 전환

**결정:** 공유 `_RateLimiter`(`mahdi/broker/rest_client.py`)의 기준 간격을
`DEFAULT_MIN_REQUEST_INTERVAL_SECONDS` 0.5초(2건/초, 2026-07-08 추정치) → 1.0초(1건/초)로
상향하고, 고정 간격 대신 **적응형**으로 바꿨다 — 500 응답 바디에서 KIS 에러코드 `EGW00201`(초당
거래건수 초과)을 감지하면 다음 호출부터 간격을 1.5배씩 넓히고(상한: 기준 간격의 4배), 그 넓어진
간격에서 연속 성공이 20건 쌓일 때마다 서서히(0.9배씩) 기준 간격까지만 되돌린다. `EGW00201`이
아닌 500(계좌 미승인 등)은 백오프 대상에서 제외했다.

**Why:** 2026-07-20 로그·DB 점검 중, `_collect_option_chain_cycle`이 행사가마다 콜→풋 순서로
호출하는데 콜은 거의 항상 성공(18~19건/8분)하고 풋만 계속 500(3건/8분)이 되는 정확한 교대 패턴을
5개 행사가 전부에서 동일하게 발견했다 — 이는 2건/초 페이싱에서 매 쌍의 두 번째 호출(0.5초 뒤)만
KIS의 실제 초당 한도를 넘긴다는 강한 정황이었다. 2026-07-08에 "2건/초"로 잡은 값 자체가 "문서화된
한도가 없어 보수적으로 추정"한 값이었을 뿐이고, 이번이 그 추정이 실측과 어긋난 두 번째 사례라
(2026-07-08: 무제한 → 2건/초, 2026-07-20: 2건/초 → 1건/초) 고정값 하나를 다시 맞히는 대신 앞으로
계좌/시간대별로 또 달라질 가능성 자체를 코드가 흡수하게 만들기로 했다. 재시작 직후 실측으로
콜:풋 비율이 18:3(~17%) → 120:75(~63%)로 개선됨을 확인했다.

**How to apply:** 앞으로 KIS REST 500이 대량 발생하면 먼저 응답 바디의 `msg_cd`를 확인할 것
(`_log_kis_call_failure`가 2026-07-19 이후 모든 주요 폴러에 응답 바디를 남기도록 표준화됨,
`option_chain`/`expiry_liquidity`/`investor_flow`/`macro_snapshot`). `EGW00201`이면 적응형
백오프가 자동으로 대응하므로 기준 간격을 또 손으로 올릴 필요는 없다 — 그래도 상한(4배)에
반복적으로 부딪힌다는 게 로그로 확인되면, 그때는 정말 기준 간격 자체를 올려야 할 신호다.

## [2026-07-10] Cross-asset stress는 스팟/지수가 아니라 CME/CBOE/HKEx 선물로, US10Y는 가격(zn_front)과 수익률(us10y_yield)을 별도 컬럼으로 분리

**결정:** VIX·USDCNH는 KIS 해외지수·환율(N/X구분) 대신 해외선물옵션 도메인의 CBOE VIX 선물(`VX`)·HKEx
USD/CNH 선물(`CNH`)로 수집한다. US10Y는 CME/CBOT 10년 국채선물(`ZN`, 선물가·역상관)과 해외주식 국채구분
일봉(`Y0202`, 실제 수익률 %)을 `macro_snapshot_5m`의 서로 다른 컬럼(`zn_front`/`us10y_yield`)에 나눠
담고, 하나로 합치지 않는다.

**Why:** 조사 결과 KIS는 VIX 현물지수 자체를 아예 제공하지 않고(관련 ETN만 있음), 환율(X구분)에는 온셰어
CNY만 있고 역외 CNH가 없다 — 즉 "지수/환율 API로 스팟값을 받는" 자연스러운 접근이 애초에 막혀 있었다.
반면 CME/CBOE/HKEx 선물은 정확히 원하는 이름 그대로(`VX`=VIX, `CNH`=USDCNH) 존재하고, VIX 근월-차근월
스프레드는 실무에서 쓰는 VIX 기간구조의 표준 정의라 오히려 더 정확하다. US10Y 쪽은 실측으로 두 경로의
성격이 다름을 확인했다: 국채구분(I) API는 계좌 제약 없이 실제 수익률 레벨을 주지만 분봉이 아예 안 되고
(`ERROR INVALID FID_COND_MRKT_DIV_CODE`), ZN 선물은 5분봉이 되지만 CBOT 거래소 신청이라는 별도 계좌
게이트가 있다(`EGW00552`) — 어느 한쪽만 쓰면 "5분 급변 감지"와 "정확한 수익률 레벨" 중 하나를 포기해야
한다. 선물가(ZN)를 억지로 수익률로 환산해 같은 컬럼에 넣으면 근사 오차가 섞이고, 반대로 수익률 컬럼에
선물가를 그대로 넣으면 단위(%  vs 가격 포인트)가 뒤섞여 하류 피처 계산이 조용히 틀어진다.

**How to apply:** 새로운 매크로 지표를 추가할 때 "KIS가 그 이름 그대로 된 상품을 주는가"를 해외주식
(지수/환율/국채) 도메인과 해외선물옵션(CME/CBOE/ICE/HKEx 등) 도메인 양쪽에서 먼저 확인할 것 — 하나가
계좌 제약(거래소별 신청)에 걸리면 다른 하나가 대체/보완 경로가 될 수 있다. 선물가 기반 컬럼과 레벨
(수익률/지수) 기반 컬럼은 이름에 단위를 드러내고(`_front`=가격, `_yield`=%) 항상 분리해서 저장할 것 —
나중에 합쳐서 피처를 계산하는 건 언제든 할 수 있지만, 한 번 섞여서 적재된 값은 사후에 단위를 되돌릴
수 없다. 해외선물 마스터(`ffcode.mst`)의 최근월물여부 플래그로 롤오버를 자동화하는 패턴
(`mahdi/data/overseas_future_master.py`)은 향후 다른 CME/CBOE/HKEx 상품을 추가할 때도 그대로 재사용
가능하다.

## [2026-07-10] 실행 중인 mahdi.main을 코드 변경 반영을 위해 재시작할 때는 별도 검증 스크립트로 토큰을 재발급받지 않는다

**결정:** `mahdi.main`이 실제로 떠 있는 상태에서 KIS 응답을 확인해야 할 때, 별도의 1회성 스크립트로
새 토큰을 발급받아 확인하지 않고 프로세스 자체를 재시작해 그 로그로 검증했다.

**Why:** `TokenDaemon`(`mahdi/broker/token_daemon.py`) 주석에 이미 경고돼 있듯, KIS는 같은 앱키로 여러
프로세스가 동시에 토큰을 발급받으면 직전 토큰을 무효화할 수 있다. 이미 실행 중인 관측 루프가 캐싱해 쓰고
있는 토큰이 있는 상태에서 별도 스크립트가 토큰을 새로 발급받으면, 그 관측 루프가 다음 REST 호출에서
401을 맞아 죽을 위험이 있다 — 진단 목적의 임시 스크립트가 운영 중인 수집 파이프라인을 망가뜨리는 건
받아들일 수 없는 트레이드오프다. 또한 KIS 토큰 발급 자체가 분당 1회 제한이 있어(2026-07-10 실측,
403 Forbidden), 짧은 간격으로 스크립트를 반복 실행하는 것도 비효율적이다.

**How to apply:** 실행 중인 프로세스가 있는 상태에서 새 코드 경로를 라이브로 검증해야 하면, 별도
스크립트를 새로 띄우기보다 "코드 반영 → 그 프로세스 자체를 재시작 → 재시작 로그/DB로 확인"을 기본
경로로 삼을 것. 재시작 전엔 항상 `docker-compose.yml`이 마운트한 `db/migrations/`가 이미 초기화된
컨테이너에는 자동 적용되지 않는다는 점([[CURRENT_STATE]] 참고)을 기억해 `docker exec -i mahdi_timescaledb
psql -U mahdi -d mahdi < 새마이그레이션.sql`을 먼저 실행할 것. 프로세스는 `uv run python -m mahdi.main
>> logs/observation_loop.log 2>&1`(사용자가 원래 쓰던 launch 방식과 동일)로 재기동해 로그 파일 하나로
이어지게 유지한다.

## [2026-07-10] 레짐 워밍업 실데이터화 — cross_asset_stress/macro_score는 근사치·스텁으로 제한, HMM fit은 오프라인 배치로 분리

**결정:** `main.py`가 매 선물봉마다 `RegimeStateMachine.step()`을 호출하도록 배선하되, 스펙(§7.3/§8)이
요구하는 입력 중 이 코드베이스에 데이터 소스가 없는 부분은 **정직하게 근사치/중립 스텁으로 제한**했다:
`cross_asset_stress`(USDKRW·USDCNH·US10Y)는 0.0 고정 반환, `macro_score`(VIX 기간구조·S&P선물 등 완전한
나침반)는 이미 수집 중인 `investor_flow_1m`의 외국인 순매수 부호로 근사. 반대로 `gap_zscore`·
`prior_close_regime`은 이미 수집 중인 실데이터(`underlying_spot_1m`/`option_analysis_1m`/`regime_state`)로
완전히 실계산했다. `RegimeEngine.fit()`은 main.py가 직접 호출하지 않고 `scripts/fit_regime_engine.py`
오프라인 배치로 분리해, `data/models/regime_engine.pkl` 존재 여부만으로 `RegimeStateMachine`이
predict()/warmup_fallback()을 자동 판단하게 했다.

**Why:** 사용자가 "완벽하게 개선 구현해"라고 요청했지만, 실제로는 세 가지 하드 제약이 있었다.
① `RegimeEngine.fit()`은 "수십 세션 분량" 피처 이력이 필요한데(regime.py 원 주석) 지금은 feature_store가
텅 비어 있어 오늘 당장 fit()해도 무의미하다 — 데이터가 쌓이는 시간 자체를 코드로 대체할 수 없다.
② USDCNH/US10Y/VIX 기간구조는 KIS API 범위 밖이라 별도 데이터 소스 연동이 필요한 별개의 큰 작업이다.
③ 완전한 macro_score도 마찬가지로 외부 매크로 데이터가 필요하다. 이 제약을 무시하고 대충 지어낸 값을
넣으면 "고쳐진 것처럼 보이지만 실제로는 근거 없는 신호"가 되어 나중에 디버깅을 더 어렵게 만든다 —
그래서 계산 가능한 부분(gap/prior_regime)은 실데이터로 완전히 배선하고, 안 되는 부분(cross_asset_stress/
macro_score)은 명시적 근사치·스텁+TODO로 남겨 "이 값은 아직 근사치다"가 코드에서 바로 보이게 했다.
EnterPlanMode로 이 범위를 사용자에게 먼저 제시하고 승인받은 뒤 구현했다.

**How to apply:** 앞으로 USDKRW/USDCNH/US10Y나 VIX 기간구조 데이터 소스를 연동하게 되면
`mahdi/features/regime_features.py`의 `cross_asset_stress()` 스텁과 `mahdi/engines/regime_pipeline.py`의
`compute_macro_score_proxy()`를 실데이터 버전으로 교체할 것 — 함수 시그니처는 유지하고 내부 구현만

**How to apply:** 새 폴링 루프를 추가할 때 "작업 끝나고 interval만큼 sleep"하는 패턴을 쓰면 이 문제가 재발한다 — `poll_time`(또는 다른 절대시각 타임스탬프)을 분/초 단위로 잘라 DB에 적재하는 루프라면 반드시 이 고정 틱 패턴(`next_tick` 누적 + `delay<0`이면 즉시 재기준, 캐치업 안 함)을 재사용할 것. 여러 폴러가 같은 `KISRestClient`(=같은 레이트리미터)를 공유하는 상태에서 폴러를 새로 추가할 때는, 기존 폴러와 정규 사이클이 반복적으로 겹치지 않도록 `startup_offset_seconds` 같은 시작 지연을 함께 검토한다. **다음 확인 필요**: 정규장 하루 운영 후 이 5분 간격 유실 패턴이 실제로 사라졌는지 DB로 재확인([[NEXT_TODO]] 참고).

## [2026-07-07] 배치파일 IF 블록 안 echo에 괄호를 그대로 쓰면 cmd.exe가 파싱 실패 — `^()` 이스케이프로 수정

**결정:** `scripts/start_mahdi_premarket.bat`의 `if not exist "%DOCKER_DESKTOP_EXE%" ( ... )` 블록 안 echo 문에서 `(%DOCKER_DESKTOP_EXE%)`처럼 괄호를 그대로 썼던 것을 `^(...^)`로 이스케이프했다.

**결정:** `poll_option_chain()`/`poll_expiry_liquidity()`의 DB 삽입 루프(`db.insert_option_analysis_1m`, `db.insert_expiry_liquidity_1m`)를 각각 레그/북 단위 `try/except`로 감쌌다. 실패 시 `conn.rollback()`을 반드시 호출한 뒤 다음 레그로 계속 진행한다.

**Why:** 2북(먼슬리+위클리) 동시 구독을 실제로 재시작해 켰더니, 얇은(거래 없는) 종목의 IV 등이 `option_analysis_1m.iv DECIMAL(8,6)`(절댓값 100 미만 제약) 범위를 넘는 값을 반환해 `psycopg.errors.NumericValueOutOfRange`가 발생했다. 이 예외가 `asyncio.gather()` 안에서 잡히지 않고 전파돼 `run_observation_loop`/`poll_expiry_liquidity`/`poll_investor_flow`까지 전부 강제 종료됐다(선물 틱 수신까지 멈춤) — 옵션 레그 하나의 데이터 이상이 전체 파이프라인을 죽인 것. `poll_option_chain`의 기존 docstring은 이미 "개별 종목 실패는 건너뛰고 계속 처리한다"고 명시했었지만, 그 원칙이 REST 조회 실패에만 적용되고 DB 삽입 실패에는 적용돼 있지 않았던 설계 공백이었다.

**How to apply:** `conn.rollback()`을 빼먹으면 안 된다 — psycopg는 실패한 트랜잭션에서 커밋 없이 다음 `execute()`를 거부하므로, rollback 없이 `continue`만 하면 그 사이클의 남은 레그가 전부 연쇄로 실패한다. 앞으로 "레그/북 단위로 실패를 흡수해야 하는" 루프를 새로 추가할 때는 REST 호출뿐 아니라 DB 삽입까지 같은 원칙(개별 실패 격리)을 적용해야 한다는 선례로 남긴다. 실운영 데이터(특히 신규 진입한 위클리처럼 검증 안 된 데이터 소스)는 예상 못 한 범위의 값을 낼 수 있다는 점도 재확인.

## [2026-07-06] VPIN은 개별 옵션이 아니라 선물(기초자산)에 적용한다

**결정:** `calculate_vpin()`(BVC 기반, 이미 구현·테스트돼 있었지만 아무도 안 부르던 함수)의 입력을 만드는 등거래량 버킷을, 지금 WS로 구독 중인 14개 개별 옵션 종목이 아니라 KOSPI200 선물(기초자산) 실시간체결가(H0IFCNT0)를 새로 구독해서 만든다.

**Why:** VPIN은 Easley-Lopez de Prado-O'Hara가 유동성이 충분한 단일 종목(전형적으로 지수선물)을 전제로 설계한 지표다. 오늘 실측한 개별 옵션 거래량은 분당 1~10계약 수준으로 얇아서, 등거래량 버킷이 완성되는 데 몇 시간씩 걸리거나 표준편차가 0에 가까워 VPIN이 대부분 0.5(중립)로 정체될 위험이 컸다. KOSPI200 선물은 옵션보다 훨씬 유동성이 높아 VPIN의 설계 의도에 맞고, 버킷이 합리적인 빈도로 완성된다.

**How to apply:** 대가로 선물 실시간체결가 WS 구독을 새로 추가해야 했다(`main.py`의 `run_observation_loop`가 옵션 ATM±3 구독과 별개로 `H0IFCNT0`/`futures_symbol`을 구독). 필드 순서가 옵션(H0IOCNT0)과 다르므로 별도 파서(`_parse_futures_tick`, `_FUT_IDX_*`)가 필요했다 — 공식 문서(시트 "지수선물 실시간체결가", API 실시간-010)로 실측 검증함. COCKPIT Flow Radar의 "가장 활발한 실제 종목 자동선택" 로직은 안 건드렸다 — 선물이 옵션보다 훨씬 자주 체결되므로 자연스럽게 대표 종목으로 뽑혀 OFI/price/microprice/VPIN이 전부 일관되게 선물 기준으로 표시된다.

## [2026-07-06] 배치파일은 CRLF 줄바꿈 필수

**결정:** Windows 작업 스케줄러용 `.bat` 파일은 반드시 CRLF(`\r\n`) 줄바꿈으로 저장한다.

**Why:** LF만 있는 배치파일을 `cmd.exe`로 실행하면 `'FILE' is not recognized...`, `&& was unexpected at this time` 같은 파싱 에러가 나며, 어느 줄이 깨지는지도 실행마다 다르게 나타난다(같은 파일인데 증상이 바뀜). 원인은 cmd.exe 배치 파서가 줄 경계 인식에 CR을 사실상 요구하기 때문. Write 도구/heredoc으로 만든 파일은 기본이 LF라 매번 변환이 필요했다.

**How to apply:** `.bat` 파일을 새로 만들거나 수정한 뒤에는 `sed -i 's/$/\r/' file.bat` 등으로 CRLF 변환 후 반드시 실행 테스트를 한다. PowerShell `Out-File`/`Set-Content`는 기본이 CRLF라 이 문제가 없다.
