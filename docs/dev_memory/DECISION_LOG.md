## [2026-07-06] (아래 결정을 뒤집음) VPIN을 옵션에도 종목 구분 없이 통일 적용

**결정:** 바로 아래 "[2026-07-06] VPIN은 개별 옵션이 아니라 선물(기초자산)에 적용한다" 결정을 뒤집는다. `run_observation_loop`의 선물 전용 특수 분기를 없애고, `aggregators`/`volume_buckets`/`vpin_returns`/`vpin_volumes`를 전부 종목별 dict로 통일해 선물이든 옵션이든 봉이 완성되면 그 종목의 VPIN을 계산해 적재한다.

**Why:** Flow Radar를 선물/옵션 두 계열로 분리한 뒤, 사용자가 "옵션 섹션에도 VPIN 차트가 빠져있다"고 지적했다. 얇은 거래량 때문에 VPIN이 자주 0.5(중립)에 머물 수 있다는 트레이드오프를 다시 설명했더니, 사용자가 "그래도 실제로 계산해서 보여달라"고 명시적으로 선택 — "일부 계열에서만 신뢰도가 낮을 수 있는 지표"보다 "화면에 아예 안 보이는 것"이 더 나쁘다는 판단으로 이해했다.

**부작용과 대응:** 옵션에도 vpin이 채워지면서, Flow Radar 분리에 쓰던 "`vpin IS NOT NULL` = 선물" 휴리스틱이 깨졌다 — `active_futures_symbol` 레지스트리 테이블(신규)로 교체해 선물 식별을 명시적으로 만들었다.

**How to apply:** VPIN처럼 "특정 조건에서만 신뢰도가 낮은" 지표를 놓고 "아예 안 보여줄지 vs 낮은 신뢰도를 감수하고 보여줄지" 갈림길에 서면, 기본값으로 후자(보여주되 한계를 설명)를 우선 제안한다 — 이번에 사용자가 그렇게 선택했다. 다만 코드 구조상 "vpin 유무로 선물/옵션을 구분"처럼 어떤 컬럼의 존재 여부에 암묵적으로 의미를 싣는 설계는, 나중에 그 컬럼을 다른 곳에도 채우게 되면 깨지기 쉽다는 교훈도 남겼다 — 애매하면 처음부터 명시적 레지스트리/플래그를 두는 편이 낫다.

## [2026-07-06] VPIN은 개별 옵션이 아니라 선물(기초자산)에 적용한다

**결정:** `calculate_vpin()`(BVC 기반, 이미 구현·테스트돼 있었지만 아무도 안 부르던 함수)의 입력을 만드는 등거래량 버킷을, 지금 WS로 구독 중인 14개 개별 옵션 종목이 아니라 KOSPI200 선물(기초자산) 실시간체결가(H0IFCNT0)를 새로 구독해서 만든다.

**Why:** VPIN은 Easley-Lopez de Prado-O'Hara가 유동성이 충분한 단일 종목(전형적으로 지수선물)을 전제로 설계한 지표다. 오늘 실측한 개별 옵션 거래량은 분당 1~10계약 수준으로 얇아서, 등거래량 버킷이 완성되는 데 몇 시간씩 걸리거나 표준편차가 0에 가까워 VPIN이 대부분 0.5(중립)로 정체될 위험이 컸다. KOSPI200 선물은 옵션보다 훨씬 유동성이 높아 VPIN의 설계 의도에 맞고, 버킷이 합리적인 빈도로 완성된다.

**How to apply:** 대가로 선물 실시간체결가 WS 구독을 새로 추가해야 했다(`main.py`의 `run_observation_loop`가 옵션 ATM±3 구독과 별개로 `H0IFCNT0`/`futures_symbol`을 구독). 필드 순서가 옵션(H0IOCNT0)과 다르므로 별도 파서(`_parse_futures_tick`, `_FUT_IDX_*`)가 필요했다 — 공식 문서(시트 "지수선물 실시간체결가", API 실시간-010)로 실측 검증함. COCKPIT Flow Radar의 "가장 활발한 실제 종목 자동선택" 로직은 안 건드렸다 — 선물이 옵션보다 훨씬 자주 체결되므로 자연스럽게 대표 종목으로 뽑혀 OFI/price/microprice/VPIN이 전부 일관되게 선물 기준으로 표시된다.

## [2026-07-06] `data_source.py`(또는 그 하위 모듈)를 고치면 COCKPIT(Streamlit) 프로세스를 반드시 재시작해야 함

**결정:** `mahdi/dashboard/app.py`가 아니라 그 아래에서 `import`되는 모듈(`data_source.py`, `panels/*.py` 등)을 수정했을 때는, COCKPIT 프로세스를 재시작해서 완전히 새로 띄운다. 프로그램적으로 넣어둔 `st.rerun()` 폴링이나 브라우저 새로고침만으로는 부족하다.

**Why:** Streamlit은 최상위 엔트리 스크립트(`app.py`)만 매 리런마다 디스크에서 다시 읽어 실행하고, `import`된 하위 모듈은 파이썬의 `sys.modules` 캐시에 남아 재사용된다. 오늘 `data_source.py`를 여러 차례(기초자산 스팟 재설계, 옵션 체인, 수급 연동) 고쳤는데도 COCKPIT 화면이 계속 "합성 리플레이" 경고를 띄웠던 사고가 있었다 — 원인은 07:30에 뜬 COCKPIT 프로세스를 하루 종일 한 번도 재시작하지 않아서, 가장 오래된(고정 라벨 `"KOSPI200_OPT"`로 조회하던) `data_source.py`를 계속 메모리에 캐싱한 채 실행하고 있었기 때문. 그 라벨은 심볼 분리 수정 이후 아무도 안 써서 09:09에 멈춘 화석 데이터였고, 그래서 항상 빈 결과 → `None` → 합성 폴백이 나왔던 것.

**How to apply:** 관측 루프(`mahdi.main`)와 COCKPIT(`streamlit`)은 서로 다른 프로세스이므로 재시작 필요 여부도 따로 판단한다 — 관측 루프 코드(`main.py`, `db.py` 등)만 고쳤으면 COCKPIT은 안 건드려도 되지만, `mahdi/dashboard/` 아래 아무 파일이나 고쳤으면 COCKPIT도 반드시 재시작한다. 원인 진단을 쉽게 하려고 `data_source.py`의 `except Exception` 폴백 분기에 `logger.exception(...)`을 추가해뒀다(예전엔 조용히 `None`만 반환해 사후 추적이 불가능했음).

## [2026-07-06] KIS WS 실시간 프레임은 파이프(|) 헤더를 벗긴 뒤에 캐럿(^) 파싱해야 함

**결정:** `mahdi.main._parse_tick`은 `raw.split("^")` 전에 `raw.split("|", 3)[-1]`로 `암호화유무|TR_ID|데이터건수` 헤더를 먼저 제거한다.

**Why:** ATM±3(콜/풋 최대 14종목) 구독 중 서로 다른 종목의 체결가가 한 봉에 뒤섞이는 버그를 고치려고 틱의 종목코드(0번 필드)를 처음 읽었더니, `market_raw_1m.symbol VARCHAR(20)`을 넘겨 실거래 중 `StringDataRightTruncation`으로 반복 크래시. 원인은 KIS WS 프레임이 `"0|H0IOCNT0|001|<실제데이터^...>"` 형태로 파이프 헤더를 앞에 붙여 오는데, 이를 벗기지 않고 전체를 `"^"`로만 쪼개면 0번 필드에 헤더 전체 + 실제 0번 필드가 합쳐져버림. idx1(BSOP_HOUR) 이후 필드는 헤더 뒤에 있어 우연히 정상 정렬됐던 것이라 지금까지 안 들켰음(가격/거래량/호가는 전부 정상으로 보였음).

**How to apply:** KIS WS 실시간 데이터를 새로 파싱하는 코드를 추가할 때는 항상 파이프 헤더 제거를 먼저 하고, 0번 필드(대개 종목코드)를 쓰기 전에는 길이/포맷을 실측으로 재확인한다. `raw.split("|", 3)[-1]`은 헤더가 없는 입력(테스트 픽스처 등)에도 안전하게 통과한다(전체 문자열을 그대로 반환).

## [2026-07-06] 배치파일은 CRLF 줄바꿈 필수

**결정:** Windows 작업 스케줄러용 `.bat` 파일은 반드시 CRLF(`\r\n`) 줄바꿈으로 저장한다.

**Why:** LF만 있는 배치파일을 `cmd.exe`로 실행하면 `'FILE' is not recognized...`, `&& was unexpected at this time` 같은 파싱 에러가 나며, 어느 줄이 깨지는지도 실행마다 다르게 나타난다(같은 파일인데 증상이 바뀜). 원인은 cmd.exe 배치 파서가 줄 경계 인식에 CR을 사실상 요구하기 때문. Write 도구/heredoc으로 만든 파일은 기본이 LF라 매번 변환이 필요했다.

**How to apply:** `.bat` 파일을 새로 만들거나 수정한 뒤에는 `sed -i 's/$/\r/' file.bat` 등으로 CRLF 변환 후 반드시 실행 테스트를 한다. PowerShell `Out-File`/`Set-Content`는 기본이 CRLF라 이 문제가 없다.

## [2026-07-06] 옵션 만기 판별은 월물구분코드가 아니라 한글종목명에서 추출

**결정:** KIS 종목코드 마스터파일(`fo_idx_code_mts.mst`)에서 옵션 콜/풋의 최근월(가장 가까운 만기) 판별은 `월물구분코드` 컬럼이 아니라 `한글종목명`(예: "C 202607   545.0")에서 정규식(`^[CP]\s*(\d{6})`)으로 뽑은 YYYYMM을 쓴다. 선물은 그대로 `월물구분코드`를 신뢰한다.

**Why:** 실제 파일을 내려받아 확인한 결과, KOSPI200 콜옵션 2664행에 대해 `월물구분코드`는 단 3개 값(1/2/3)만 존재하는데 실제 만기월은 11개(202607~202812)였다 — 즉 이 컬럼은 옵션에서는 만기 순번이 아니라 다른 분류(유동성 등급 추정)를 나타낸다. 선물은 7개 행에 월물구분코드 1~7이 만기 순서와 정확히 일치해 문제없이 사용 가능.

**How to apply:** `symbol_master.py`의 `options()`/`nearest_expiry_chain()`/`option_symbol()`은 모두 `한글종목명` 파싱 결과(`만기YYYYMM` 컬럼)를 기준으로 동작한다. 새로운 조회 메서드를 추가할 때도 옵션 만기 필터링에는 절대 `월물구분코드`를 쓰지 않는다.

## [2026-07-06] KIS 종목코드 마스터파일 실제 컬럼 순서는 공식 참고 스크립트와 다름

**결정:** `symbol_master.py`의 `_COLUMNS`는 KIS 공식 GitHub 참고 스크립트(`domestic_index_future_code.py`)가 명시한 순서(`...,ATM구분,행사가,월물구분코드,...`)가 아니라 실측 순서(`...,월물구분코드,행사가,ATM구분,...`)를 따른다.

**Why:** 실제 파일을 내려받아 raw 필드를 직접 `split('|')`해서 확인한 결과, 동일 만기의 여러 행사가에서 5번째 필드가 계속 '2'로 고정되고(=월물구분코드류) 6번째 필드가 행사가별로 다르게 변하는(=행사가) 패턴을 확인. 참고 스크립트의 컬럼명 순서를 그대로 믿었다가 처음엔 만기/행사가가 전부 어긋난 체인이 나왔다.

**How to apply:** KIS가 제공하는 참고 스크립트/문서는 "출발점"으로만 쓰고, 실제 파일을 받아 값을 확인하기 전에는 신뢰하지 않는다. 마스터파일 포맷이 KIS 쪽에서 바뀌면(가능성 낮지만) `symbol_master.py`의 `_COLUMNS` 주석에 있는 실측 근거부터 다시 검증해야 한다.

## [2026-07-06] 지수옵션 실시간 시세(WS)는 모의투자 전용 도메인이 없음

**결정:** `H0IOCNT0`/`H0IOASP0`(지수옵션 실시간체결가/호가) 구독은 계좌가 모의투자든 실전이든 항상 `ws://ops.koreainvestment.com:21000`(실전 도메인)에 접속한다. `is_mock` 여부로 도메인을 분기하지 않는다.

**How to apply:** `tr_codes.py`의 `MARKET_DATA_WS_DOMAIN`은 시세 구독 전용으로 항상 실전 도메인을 가리킨다. Phase2에서 주문체결통보(계좌별 알림)를 붙일 때는 `WS_TR_ORDER_NOTICE`처럼 `is_mock` 기준으로 도메인/TR_ID를 분기해야 한다 — 이 둘을 같은 로직으로 착각하지 않는다.

## [2026-07-05] .gitignore의 `data/`는 `/data/`로 고정

**결정:** `.gitignore`의 `data/` 패턴을 `/data/`(저장소 루트 고정)로 수정.

**Why:** 고정하지 않은 `data/`는 깊이에 상관없이 이름이 "data"인 모든 디렉터리를 무시한다 — 결과적으로 `mahdi/data/`(실제 소스 패키지: collector.py/db.py/subscription_manager.py)가 첫 커밋부터 git에 전혀 안 잡히고 있었다.

**How to apply:** 앞으로 `.gitignore`에 일반적인 이름(data, build, dist, cache 등)의 디렉터리를 추가할 때는 의도한 위치(보통 루트)에만 적용되도록 항상 `/` 접두사로 고정한다. 새 gitignore 규칙을 추가한 뒤에는 `git status`로 예상치 못하게 무시되는 소스 파일이 없는지 확인한다.

## [2026-07-05] Regime Engine 테스트는 실제 HMM 적합이 아니라 stub 모델로 검증

**결정:** `RegimeEngine`의 라벨 매핑/확률벡터 재정렬/상위 레짐 충돌 해소 로직은 가짜(stub) 모델을 주입해 결정론적으로 테스트하고, 실제 `GaussianHMM.fit()`은 느슨한 어서션의 스모크 테스트로만 확인한다.

**How to apply:** 새로운 통계적/수치적 적합(fit) 로직을 테스트할 때는 "우리가 작성한 로직"(라벨 매핑, 후처리)과 "라이브러리의 수치 최적화 결과"를 분리해서, 후자는 구조적 검증(예외 없음, 확률합=1)만 하고 전자는 stub으로 결정론적으로 검증한다.
