## 1. 시스템 설계 목표 상세 (Detailed Strategic Goals)

### 1.1. 금융공학 기반의 수익화 (Financial Engineering & Alpha)
단순한 방향성 추종자가 아닌, 시장의 구조적 빈틈을 공략하는 **'포식자'** 모델을 구축합니다.
*   **변동성 비대칭성 (Volatility Asymmetry)**: 하락에 대한 공포로 인해 발생하는 풋 옵션의 고평가와 콜 옵션의 저평가 괴리($Skew$)를 실시간으로 추적합니다. 내재변동성($IV$)과 실현변동성($RV$)의 차이인 변동성 리스크 프리미엄(VRP)을 극대화하는 구간에서만 진입합니다.
*   **시장 미시구조 (Market Microstructure)**: VPIN(Volume-Positive Informed Trading) 알고리즘을 통해 거래량과 가격 변화 속에서 '정보를 가진 세력'의 주문 흐름 독성(Order Flow Toxicity)을 감지합니다. 
*   **감마 노출도 (GEX) 분석**: 시장 전체의 감마 노출 수준을 계산하여 지수가 특정 가격대에 도달할 때 발생할 수 있는 가속화(Gamma Squeeze) 지점을 선제적으로 장악합니다.

### 1.2. 하이브리드 청산 시스템 (Hybrid Liquidation)
기계의 속도와 인간의 통찰력을 결합한 다층적 탈출 전략입니다.
*   **지능형 손절 (Dynamic Stop)**: 고정 손절이 아닌 ATR(평균 실질 변동폭)의 $n$배수를 적용하여 시장 노이즈에 의한 강제 퇴출을 방지합니다.
*   **상태 기반 청산**: 수익이 발생하더라도 앙상블 모델의 예측 등급이 하락(A → C)하거나, 선물 베이시스가 역전될 경우 '이익 보존 청산'을 즉시 실행합니다.
*   **하이브리드 오버나이트**: 15:10분 활성화. 시스템이 승률 통계와 미국 야간 지표 위험도를 계산하여 제안하면, 트레이더가 최종 승인하는 '검문소' 시스템을 운영합니다.

### 1.3. 자기 강화 학습 (Self-Reinforcement Learning)
경험을 지능으로 바꾸는 **'바둑 AI'** 방식의 학습 구조입니다.
*   **실시간 피드백 루프**: 1분 전의 예측이 1분 후의 결과와 얼마나 일치했는지 즉시 채점($Loss\ Calculation$)하고, SGD(확률적 경사 하강법)를 통해 모델 가중치를 실시간 업데이트합니다.
*   **보상 설계 (Reward Function)**: 단순히 수익률뿐만 아니라 샤프 지수(Sharpe Ratio)와 승률을 결합한 다중 보상 체계를 통해 '안정적인 수익 곡선'을 지향하도록 학습합니다.

### 1.4. 챔피언-도전자 프레임워크 (Champion-Challenger)
전략의 노후화를 방지하는 **'전략 서바이벌 리그'**입니다.
*   **리그제 운영**: 현재 자금을 집행하는 '챔피언' 전략과 백그라운드에서 가상 매매를 수행하는 '도전자' 그룹을 상시 가동합니다.
*   **자동 승격 시스템**: 주간 단위 평가에서 도전자의 샤프 지수가 챔피언을 일정 비율 이상 상회할 경우, 시스템은 자동으로 전략 엔진을 교체합니다.

3. min_samples_leaf 란

매우 중요합니다.

뜻:

리프 하나에 최소 몇 개 데이터가 있어야 하는가

입니다.

예시

데이터 1개만 있어도
새 leaf 생성 가능

문제:

너무 예민
과최적화(overfitting)

발생 가능.

min_samples_leaf = 50
최소 50개 데이터 있어야
새 leaf 생성

즉:

더 둔감
안정적
노이즈 감소
중학생식 비유

학생 성적 예측한다고 생각.

너무 작은 값
학생 1명만 보고
"이 학교 천재학교!"

판단.

→ 위험.

적당한 값
50명 이상 데이터 보고 판단

→ 더 안정적.

상수화(constantization)란

예전 코드:

model = LGBMClassifier(
    min_samples_leaf=37
)

이렇게 여기저기 흩어져 있으면 관리가 어렵습니다.

상수화 후
MIN_SAMPLES_LEAF = 37

그리고:

    min_samples_leaf=MIN_SAMPLES_LEAF
)

처럼 사용.

왜 상수화하나
이유 1 — 유지보수

값 변경 쉬움.

이유 2 — 실험 관리

예:

현재 운영값 = 32
실험값 = 64

관리 가능.

이유 3 — 전략 일관성

모든 모델 동일 기준 적용.
